## How to execute a ticket

### Before starting
1. Read the story file: `docs/tickets/<ID>-<slug>.md`
2. Confirm all `depends_on:` IDs are `status: done`.
3. Read the relevant instruction files:
   - C++ core code → `.github/instructions/cpp-hot-path.instructions.md`
   - Python tooling → `.github/instructions/python-tooling.instructions.md`
   - Architecture questions → `.github/instructions/tracking-architecture.instructions.md`
4. Move ticket to in-progress: `python tools/board/ticket_move.py <ID> in-progress --note "Starting execution"`

### During execution
- Follow the **Plan** section's U-ID steps sequentially.
- Each U-ID step is one logical unit of work (one function, one file, one test).
- Run tests after each step where applicable.
- If a step reveals a design question → check ADRs in `Claude Synthesised/` first.

### After completing
1. Run all relevant tests (`cmake --build . --target test`, or `pytest`).
2. Move ticket to done: `python tools/board/ticket_move.py <ID> done --note "All acceptance criteria met"`
3. Verify with board lint: `python tools/board/board_check.py`

## Design-tier tickets: how to handle sub-tickets

Design-tier tickets (TRK-009, TRK-020, TRK-025) have **child tickets** (TRK-009a/b/c/d, etc.):
- The parent story file has a sub-ticket table + orchestration notes.
- Execute child tickets independently following their own `depends_on:` chains.
- The parent is "done" when all children are done.

## Resuming after interruption

If you're picking up mid-way through v0.3:
1. Run `python tools/board/board_check.py` to verify board consistency.
2. Look at `BOARD.md` → "In Progress" and "Done (recent)" to see current state.
3. For each `done` ticket, its story file confirms completion in the `## Log` section.
4. Find the next ready ticket using the `depends_on` filter logic above.
5. Continue from there — no other context needed.
